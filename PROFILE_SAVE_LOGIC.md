# 用户档案手动保存逻辑说明 (Profile Save Logic)

## 1. 概述
前端已将用户档案保存模式从“自动同步”更改为“手动保存”。用户在编辑完档案信息或拍摄照片后，必须点击页面底部的 **“保存档案”** 按钮，前端才会向后端发送 `POST` 请求。

## 2. 交互流程
1.  **用户操作**: 用户进入“医美档案”页面，修改性别、年龄、添加医美经历，或点击头像进入相机页面拍照。
2.  **暂存状态**: 所有修改仅保存在前端内存 (`React State`) 中。
3.  **触发保存**: 用户点击页面底部悬浮的 **“保存档案”** 按钮。
4.  **网络请求**: 前端调用 `dataService.saveUserProfile`，向后端发送数据。
5.  **反馈**:
    *   **成功**: 弹出 "保存成功" 提示 (Toast)，2秒后自动消失。
    *   **失败**: 弹出 `alert` 警告框提示失败原因。

## 3. 接口详情

### 3.1 保存接口
*   **URL**: `POST /user/profile`
*   **Content-Type**: `application/json`

### 3.2 请求体结构 (Request Body)
前端发送的 JSON 数据结构如下：

```json
{
  "user_id": "user_123456",  // 前端生成的唯一用户ID (存储在 localStorage)
  "data": {
    "gender": "女",          // 字符串: "男" | "女" | ""
    "age": "25",            // 字符串或数字
    "history": [            // 数组: 历史项目列表
      "热玛吉",
      "光子嫩肤"
    ],
    "photo": "data:image/jpeg;base64,/9j/4AAQSkZJRg..." // 字符串: Base64 格式的图片数据 (若未拍照则可能为 null 或旧 URL)
  }
}
```

> **注意**: `photo` 字段如果是新拍摄的照片，将以 `data:image/jpeg;base64` 开头的 Base64 字符串形式发送。后端需要支持接收较大的 Base64 字符串并在服务器端将其保存为图片文件（如需），或直接存储在 JSON 中（不推荐存大图在 JSON，建议后端提取保存并替换为 URL）。**当前协议中，后端仅需将整个 `data` 对象保存为 `profile.json` 即可。**

## 4. 特殊场景说明
*   **拍照返回**: 用户从相机页面拍照返回后，照片数据 (`newPhoto`) 会通过路由参数传递给 Profile 页面并更新状态，但**不会自动上传**。用户需再次确认其他信息无误后，点击“保存档案”一并提交。
*   **加载逻辑**: 页面初始化时会调用 `GET /user/profile` 回显数据。如果用户刚拍完照返回（`location.state` 有数据），则优先显示刚拍的照片和刚填写的表单，而不覆盖为云端旧数据。

## 5. 后端建议
*   **数据大小**: 由于 `photo` 可能是 Base64 字符串，请求体可能较大 (2MB+)，请确保 Nginx 或后端框架 (FastAPI) 允许较大的 Request Body。
*   **幂等性**: 多次点击保存按钮会发送多次请求（前端有 `isSaving` 状态防止短时间重复点击，但网络延迟时仍可能发生），后端直接覆盖保存即可。
